## 19:39 架构自检：chat.html 链路尚未转为 OpenClaw 独立 strategy-orchestrator Agent

### 核查范围
- `chat.html`
- `server.js`
- `python_wrapper/live_agent_server.py`
- `agents/strategy-orchestrator/protocols/task_protocol.py`
- `agents/strategy-orchestrator/executors/orchestrator.py`

### 核查结论
- 网页端 `chat.html` 将用户输入通过 `POST /analyze_sse` 发送到本地 Node 代理。
- `server.js` 将 `/analyze_sse` 转发到 `LIVE_AGENT_API`，默认 `http://127.0.0.1:8003/analyze_sse`。
- `live_agent_server.py` 的 `AnalyzeRequest` 只有 `question/time_range/analysis_type/max_cycles`，没有 `session_id/chat_id/history/messages`。
- `live_agent_server.py` 仍通过 `from executors.orchestrator import create_orchestrator` 创建本地 Python orchestrator，并执行 `orchestrator.execute(task)`。
- 当前网页链路没有通过 OpenClaw `sessions_send(agentId="strategy-orchestrator", ...)` 转发给独立的 strategy-orchestrator Agent。

### 影响范围
- 浏览器历史对话只保存在前端 `localStorage.marketChatHistory`，不进入后端任务上下文。
- 每次网页发送分析请求，本质是一次独立后端任务，不具备服务端会话记忆。
- 多个浏览器窗口共享同一 origin 的 `localStorage` 历史，但后端并不知道这些窗口是不是同一个会话。
- 这与 2026-06-24 新增的任务路由规则存在偏差：复杂业务分析应转发到独立 OpenClaw Agent `strategy-orchestrator`，而不是本地 Python 调度对象。

### 修复优先级
- P0：将 `live_agent_server.py` 的复杂分析入口改为 OpenClaw `sessions_send(agentId="strategy-orchestrator", ...)`，本地 FastAPI 只做 HTTP/SSE relay。
- P0：明确新增或透传 `session_id` / `chat_id` / `conversation_summary`，避免前端历史只停留在 localStorage。
- P1：前端每个新对话生成稳定 `chat_id`，并在请求体中传给后端；同一窗口继续同一 `chat_id`。
- P1：多窗口 localStorage 同步和并发写入需要处理，避免历史覆盖。
